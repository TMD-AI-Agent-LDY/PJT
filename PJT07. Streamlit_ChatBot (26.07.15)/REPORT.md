# REPORT

## 1. Agent 활용 과정

- README 요구사항을 읽고 SQLite 기반 채팅방/메시지 구조를 먼저 설계했다.
- 이후 채팅방별로 모델과 시스템 프롬프트를 고정 저장하는 방향으로 구조를 확장했다.
- Streamlit 재실행 특성 때문에 화면 상태는 `st.session_state`로 관리하고, 실제 데이터는 SQLite에 저장하도록 정리했다.

## 2. 새롭게 배운 내용

- 채팅방 설정이 방마다 달라지면, 설정값도 방 테이블에 같이 저장하는 것이 자연스럽다.
- Streamlit에서는 입력값과 DB 값이 섞이지 않도록 현재 선택된 채팅방의 설정을 명확히 읽어와야 한다.
- 삭제 전 확인은 체크박스와 비활성화 버튼 조합으로 구현하면 사용자 실수를 줄일 수 있다.

## 3. 프로그램 동작 원리

- 앱 시작 시 SQLite 테이블이 없으면 자동 생성하고, 기존 데이터베이스가 있으면 컬럼을 보정한다.
- `chat_rooms`에는 채팅방 이름, 모델, 프롬프트, 생성 시각이 저장된다.
- `messages`에는 채팅방별 대화 내용이 저장된다.
- 사용자는 채팅방을 만들 때 모델과 프롬프트를 함께 고르고, 이후 해당 방에서는 그 설정이 고정된다.
- 메시지를 입력하면 선택된 채팅방의 모델과 시스템 프롬프트를 사용해 Groq API에 전달한다.

## 4. 오류 해결 및 개선 과정

- 모델 선택과 시스템 프롬프트를 전역 설정처럼 쓰던 구조를 바꾸고, 채팅방 단위 설정으로 옮겼다.
- 채팅방 삭제는 확인 체크 후에만 동작하도록 만들어 실수 삭제 위험을 줄였다.
- 오래된 채팅방과 최근 채팅방 정렬을 선택할 수 있게 해서 목록 탐색을 편하게 했다.
